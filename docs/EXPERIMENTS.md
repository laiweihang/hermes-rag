# RAG 增强功能实验指引

围绕赫尔墨斯本期新增的四个增强功能 ——「**简单重写**」「**HyDE 假设性答案**」「**上下文感知分块**」「**LLM 精排**」—— 设计的可手工复现实验。每个实验都给出"开关怎么切""问什么""在哪个 UI 字段看到生效证据"。

> 本文档与 [`docs/EVAL.md`](EVAL.md)（基础检索批量评测）和 [`docs/DEMO.md`](DEMO.md)（整体演示流程）互为补充：本文聚焦 **单个增强功能在前后切换时检索结果如何跳变**，靠人肉对比 UI 字段，不跑批量指标。

---

## 一、实验前置（约 5 分钟）

### 1. 启动服务并登录
1. 仓库根目录执行 `.\scripts\start-dev.ps1`（或 `bash scripts/start-dev.sh`）
2. 浏览器访问 `http://localhost:3000`，使用 `admin` / `123456` 登录

### 2. 上传教学语料

在 **文档管理** 页（侧栏 `/documents`）依次上传：

- `demo/teaching/01_hr_handbook.md`
- `demo/teaching/02_finance_q1.md`
- `demo/teaching/03_meeting_minutes.md`

> 每篇均为 markdown，不走 OCR。入库完成后列表应显示三条记录、总切片数 6-10。

### 3. 重置开关到"全关基线"

进 **管理后台** `/admin` → **检索调参** Tab，确认初始状态为：

| 开关 | 状态 | 备注 |
|---|---|---|
| 启用混合检索（BM25 融合） | 开 | 保留默认融合策略，便于观察 BM25 / 语义 rank |
| 模式 | weighted | α=0.5（语义/BM25 各半） |
| 启用重排（rerank） | **关** | 实验 4 才打开 |
| 启用上下文感知分块 | **关** | 实验 3 才打开 |
| 启用简单重写 | **关** | 实验 1 才打开 |
| 启用 HyDE 假设性答案 | **关** | 实验 2 才打开 |

> 如果之前实验残留，把上述 4 个增强开关全部置 OFF，再点页面下方"保存"按钮。

### 4. 记录基线锚点（可选）

进 `/vectors` 页搜索关键词「**1.5 倍**」，应能看到来自 `01_hr_handbook.md` 的某条 chunk。点开它，记下 `metadata` 区里有哪些字段——此时 **不应有** `chunk_context`、`contextual_chunked`、`original_text` 三个字段。等做完实验 3 后再回来对比。

## 二、实验 1：简单重写（Simple Query Rewrite）

### 目标
让 LLM 把口语化短查询补充为更接近文档语言的查询，提升 BM25 命中率。

### 操作

1. 管理后台 → 检索调参 → 找到 **「查询重写」** 区块
2. 把 **「启用简单重写」** 切到 ON
3. **不要** 同时开启 HyDE
4. 在下方 **「查询重写 Provider」** 选择一个可用 LLM Provider（与默认问答 Provider 相同即可）
5. 点击页面下方 **保存**

### 触发问句

进首页 `/` → **新建对话** → 选择 **通用问答** 技能 → 依次提问：

| 问题 | 期望 LLM 重写为 |
|------|----------------|
| 「加班费咋算」 | 「加班费的计算方法和倍率」之类 |
| 「病假咋请」 | 「病假申请流程及所需材料」之类 |

### 观察点

1. **QueryRewritePanel 出现**：助手消息正文上方应出现一个带 ✨ 图标的小卡片（点击 "展开" 可看完整重写文本），字段 `simple` 应有值，`hyde` 应为 `null`
2. **参考来源命中**：展开助手消息底部的"参考来源"折叠面板，应包含 `01_hr_handbook.md` 的第十七条 / 第二十条段落
3. **关 vs 开对比**：
   - 关闭"启用简单重写"开关 → 保存 → 在同一对话再问一次「加班费咋算」
   - QueryRewritePanel 不再出现
   - 对比两次"参考来源"的片段，注意是否漏掉了 1.5/2.0/3.0 倍率段（关闭后更可能漏）

### 客观对比（推荐做一次）

> 关键说明：`/api/admin/retrieval/preview` **不走查询重写**（直接用原始 query 调 `hybrid_search`）。要做客观对比，必须把 chat 页 QueryRewritePanel 里的重写文本 **手动复制** 到 preview 输入框。

1. 在 chat 触发"加班费咋算"，从 QueryRewritePanel 抄出 `simple` 字段文本
2. 切到管理后台 → 检索调参 → 右侧 **检索预览** 面板
3. 第一次：粘贴原始查询「加班费咋算」→ 搜索 → 截图记下 top-5
4. 第二次：粘贴重写后的查询 → 搜索 → 对比 top-5 排序
5. 重写后含倍率细则的 chunk 应排得更前，BM25 rank 上升尤其明显

## 三、实验 2：HyDE 假设性答案

### 目标
让 LLM 写一段约 100-200 字的"假答案"代替短查询做向量检索，缩短语义距离。

### 操作

1. 把 **「启用简单重写」** 关闭（避免与 HyDE 通道叠加）
2. 把 **「启用 HyDE 假设性答案」** 切到 ON
3. 共用的 Provider 选择保持不变
4. 保存

### 触发问句

新建对话 → 提问：

| 问题 | 期望 HyDE 扩写要点 |
|------|------------------|
| 「SaaS 毛利率多少」 | 应包含「Q1 财报」「SaaS 业务」「毛利率」「百分比」等正式财报用词 |
| 「应收账款周转」 | 应包含「天数」「同比改善」「营运资金」等术语 |

### 观察点

1. **QueryRewritePanel 显示长段 hyde**：展开后 `hyde` 字段应是一段约 100-200 字的"假财报段落"，`simple` 字段为 `null`
2. **参考来源命中**：应直接命中 `02_finance_q1.md` 的"二、业务线毛利分析"或"四、运营效率指标"段
3. **答案应用 65.2%、58 天**等具体数字

### 客观对比

1. 在 chat 触发后，把 QueryRewritePanel 的 `hyde` 字段（一段长文本）整段复制
2. 检索预览：先用原始查询「SaaS 毛利率多少」搜索 → 记下 `02_finance_q1.md` 含 65.2 段的 **语义 # 排名**
3. 再用 HyDE 文本搜索 → 重新观察含 65.2 段的语义 # 排名
4. **预期**：HyDE 文本下，含 65.2 段的"语义 #"会前移至前 1-3 名

### 关键说明

- HyDE 文本 **只走向量通道**，BM25 仍用原查询（参见 [`query_rewrite.py:177`](../query_rewrite.py) `apply_query_rewrite_if_enabled` 的通道分流逻辑）
- 因此 HyDE 对纯关键词类查询提升有限；它的优势在 "短查询/隐含语义" 场景

## 四、实验 3：上下文感知分块（Contextual Chunking）

### 目标
入库时 LLM 为每个切片生成 50-100 字定位上下文，前置到 page_content 后再嵌入，让丢失了上文标题的中段也能被命中。

### 重点提示

> 本功能在 **入库时** 生效，**仅切开关无效**。每次切换该开关后，相关文档必须 **删除并重新上传** 才能重新切片+重新嵌入。

### 操作流水线

#### 第 1 步：基线确认（contextual 关闭）

1. 当前 4 个增强开关全部关闭
2. `/vectors` 页搜索 `1.5 倍`，定位含倍率细则的 chunk
3. 点击该 chunk 的"详情/编辑"图标，查看 `metadata`：**应没有** `chunk_context` 字段
4. 切到管理后台 → 检索预览 → 输入「**工作日加班几倍工资**」→ 搜索 → 记下 top-5（重点看含 1.5 倍的那段排第几）

#### 第 2 步：开启上下文感知分块 + 重新入库

1. 管理后台 → 检索调参 → **「启用上下文感知分块」** 切到 ON → 选择 LLM Provider → 保存
2. 文档管理页 → 找到 `01_hr_handbook.md` → **点击删除**
3. 再次上传同一文件 `demo/teaching/01_hr_handbook.md`
4. 等待入库完成。后端日志（`logs/api.log`）会出现类似：
   ```
   🧠 启动上下文感知分块: N 个切片 / 1 个文档 / 并发 4 路
   ✅ 上下文感知分块完成: N/N 成功
   ```

#### 第 3 步：观察上下文是否真的写入向量库

1. `/vectors` 页再搜 `1.5 倍`，定位同一段
2. 点开看 `metadata`，**应新增三个字段**：
   - `chunk_context`：50-100 字摘要，类似「本段属于第十七条加班补贴的具体支付倍率部分，列出工作日 1.5 倍、休息日 2.0 倍、法定节假日 3.0 倍标准」
   - `original_text`：未加前缀的原始切片（用于显示）
   - `contextual_chunked: true`

#### 第 4 步：复跑同一查询对比

1. 检索预览输入「**工作日加班几倍工资**」→ 搜索
2. 与第 1 步的 top-5 对比。**预期**：含 1.5 倍的那段 BM25 # 与最终融合分都明显前移——因为 LLM 加的前缀里出现了「加班补贴」「倍率」等关键词
3. 也可换 `cc2` 数据集中的「**法定节假日加班几倍**」做二次验证

### 关键说明

- `page_content` 在前端"参考来源"中只展示 `original_text`，避免上下文摘要污染人眼阅读
- 重新入库的过程会消耗 LLM 调用次数（每切片 1 次）；对 `01_hr_handbook.md` 大约 4-5 次调用，几秒内完成

## 五、实验 4：LLM 精排（Rerank）

### 目标
对 `hybrid_search` 的 top-N 候选再调一次 LLM 让它打 0-10 分重排，把"形式相关"挤下去、"真正相关"提上来。

### 操作

1. 不动其他开关（contextual 可保持开/关都行；为隔离影响建议先关掉，让对比更纯）
2. 把 **「启用重排（rerank）」** 切到 ON
3. **「精排保留数」** 保留默认 5
4. 保存

### 触发查询（直接用 admin 检索预览，不走 chat）

| 查询 | BM25 假相关陷阱 | 精排预期重排 |
|------|----------------|-------------|
| 「加班政策决议」 | T04/T06/T07 三条待办（加班政策宣贯、加班补课、加班调度）BM25 易拉前列 | 决议一段排到 #1，三条待办 rerank_score 仅 2-4 |
| 「V2.0 P0 议题有哪些」 | 多条决议/待办里也含 P0 字样 | 真正的"议题 1/议题 2"清单段排到 #1 |

### 操作步骤

1. 管理后台 → 检索预览面板 → 输入「**加班政策决议**」
2. **第一次**：先确认"启用重排"是 ON，搜索；记下 top-5 排序及每条的"精排 X.X 分 (原#N)"标签
3. **第二次**：暂时把"启用重排"切到 OFF → 保存 → 同一查询重搜；这次结果不再出现"精排 X.X 分"标签，只有"语义 #"+"BM25 #"+"融合"
4. **对比要点**：
   - 关 rerank：top-3 大概率出现 T04/T06/T07 那种"加班待办"类 chunk
   - 开 rerank：top-1 应该是含「**决议一**」字样的整段决议；T04/T06/T07 退到 top-5 之后或被挤出

### 观察精排分细节

开启重排后，每条结果尾部会显示如下样式：

```
精排 9.0 分 (原#3)
精排 7.0 分 (原#1)
精排 3.0 分 (原#2)
```

- `精排 X.X 分`：LLM 给的相关性评分（0-10）
- `(原#N)`：未精排前的原始排名
- 如果出现 `(原#3)` 排在第 1 位，说明 rerank 把第 3 名提到了第 1 名 —— 这就是精排"翻盘"的直接证据

### 失败回退

LLM 调用失败或解析评分失败时，rerank 会静默回退原排序（参见 [`reranker.py:78`](../reranker.py)），不阻塞问答；可在 `logs/api.err.log` 看到 warning。

## 六、附录 A：怎么对照看每个功能"是否真的生效"

| 功能 | 在哪看证据 | 期望字段/外观 |
|------|-----------|--------------|
| 简单重写 | chat 助手消息上方 QueryRewritePanel | `simple` 字段非空、`hyde` 为 null |
| HyDE | 同上 | `hyde` 字段是一段 100-200 字文本、`simple` 为 null |
| 上下文感知分块 | `/vectors` 页 chunk 详情的 metadata | `chunk_context` / `original_text` / `contextual_chunked: true` |
| LLM 精排 | 管理后台检索预览每条结果尾部 | `精排 X.X 分 (原#N)` 标签 |

## 七、附录 B：组合实验（可选 5 分钟）

把多个开关同时打开，观察叠加效果：

1. 同时开 **简单重写 + HyDE**：BM25 通道用 simple，向量通道用 HyDE（参见 [`query_rewrite.py:240`](../query_rewrite.py) 通道分流），两路各自最优
2. 同时开 **上下文感知分块 + 重排**：先用 contextual 让 BM25/语义召回更准，再让 rerank 微调
3. **全开**：在第 2 项基础上叠加查询重写——这是日常生产推荐配置

> 注意：每开启一个功能都意味着多调一次 LLM。全开后单轮问答的延迟会增加 2-4 秒（具体取决于 LLM 速度）。生产环境建议根据知识库规模和延迟预算逐项打开。

## 八、附录 C：常见排查

| 症状 | 排查方向 |
|------|---------|
| QueryRewritePanel 不出现 | 后端是否升级了带 `[REWRITE]` SSE 事件的版本？开关是否选了可用 Provider？查 `logs/api.log` 看是否有 `🔁 simple 重写` 日志 |
| 重新上传后 chunk_context 仍为空 | 上下文感知分块开关是否在重新上传**之前**已保存？Provider 是否可达？查 `logs/api.log` 看 `🧠 启动上下文感知分块` 日志 |
| 精排分全部相同 | LLM 输出可能没按 `#1: 8` 格式；查 `logs/api.err.log` 看是否有 `rerank 无可解析评分` warning |
| 检索预览返回空 | 三篇教学语料都上传了吗？/vectors 页能看到切片吗？ |
