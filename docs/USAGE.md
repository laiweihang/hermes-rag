## 一、检索调参

赫尔墨斯的 RAG 检索层支持四种融合策略和多维可调参数。通过调参，可以针对不同类型的文档和问题风格优化检索效果。

### 1.1 前端 UI 调参（推荐）

系统在管理后台内置了**可视化检索调参面板**，无需写代码即可完成参数调优。

#### 进入方式

1. 用管理员账号（默认 `admin` / `123456`）登录系统
2. 点击侧栏「管理后台」进入 `/admin` 页面
3. 点击顶部的 **「检索调参」** 标签页

#### 界面说明

页面分为左右两栏：

**左栏 — 参数配置面板**

| 区域 | 说明 |
|------|------|
| 融合策略 | 四选一：加权融合 / RRF 排名 / 纯语义 / 纯关键词 |
| Alpha 权重 | 加权融合模式专属：滑块调节语义 vs 关键词的比例 |
| RRF 常数 k | RRF 模式专属：值越大，两路排名差异越小 |
| 召回数 | BM25 召回数 / 向量召回数 / 最终保留数 |
| 语义相关度阈值 | 低于此分的片段被丢弃，0 表示不过滤 |
| BM25 开关 | 关闭后仅走语义向量检索 |
| Rerank 开关 | 启用后，召回的片段会经过 LLM 二次打分 |
| 精排保留数 | Rerank 后最终保留给 LLM 的片段数 |

操作按钮：
- **保存到全局**：将当前参数写入数据库，后续所有对话都使用新参数
- **重置**：还原到上次保存的参数

**右栏 — 检索预览面板**

1. 在输入框填写一个测试问题
2. 点击搜索按钮（或按 Enter）
3. 系统会使用左栏的参数运行一次真实检索（**不调用 LLM**）
4. 展示每个召回片段的：
   - 来源文件名
   - 内容预览（前 200 字）
   - 融合分数
   - 语义排名 + BM25 排名
   - 精排分数（若启用 Rerank）

#### 典型调参流程

```
1. 先上传几份文档 → 确保知识库不为空
2. 在预览栏输入一个你知道答案的问题
3. 观察「命中片段」是否包含正确答案
4. 切换不同融合策略，对比哪种效果最好
5. 微调 alpha / 召回数 / 阈值，直到满意
6. 点击「保存到全局」生效
```

### 1.2 命令行批量评测

适合系统性对比多种策略的检索效果，产出量化指标（hit@1 / hit@5 / MRR）。

#### Step 1：准备评测数据集

在 `evals/` 目录下创建 `.jsonl` 文件（可参考 `evals/dataset.example.jsonl`），每行一条：

```json
{"id": "q1", "question": "加班费按多少倍计算", "expected_sources": ["hr.md"], "expected_keywords": ["1.5", "倍"]}
{"id": "q2", "question": "出差补贴每天多少钱", "expected_keywords": ["200"]}
```

字段说明：

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 唯一标识，如 `q1` |
| `question` | 是 | 测试问题 |
| `expected_sources` | 至少填一个 | 期望命中的文件名列表（不含路径） |
| `expected_keywords` | 至少填一个 | 期望出现在召回片段中的关键词 |

#### Step 2：运行评测

```powershell
cd "项目根目录"

# 默认参数（weighted + rerank）
uv run python scripts/eval_retrieval.py

# 对比不同策略
uv run python scripts/eval_retrieval.py --mode semantic --no-rerank
uv run python scripts/eval_retrieval.py --mode bm25 --no-rerank
uv run python scripts/eval_retrieval.py --mode weighted --alpha 0.7 --no-rerank
uv run python scripts/eval_retrieval.py --mode rrf --rerank --top-k 5

# 使用自定义数据集
uv run python scripts/eval_retrieval.py --dataset evals/my_dataset.jsonl
```

#### Step 3：查看结果

终端输出：

```
id       hit@1  hit@5  first_rank question
--------------------------------------------------------------------------------
q1       ✓      ✓      1          加班费按多少倍计算
q2       ·      ✓      3          出差补贴每天多少钱
q3       ✓      ✓      1          年假有多少天
--------------------------------------------------------------------------------
summary: n=3  hit@1=66.67%  hit@5=100.00%  MRR=0.8333
```

完整 JSON 结果自动保存到 `evals/results/<时间戳>_<mode>_<rerank>.json`。

#### 评测脚本完整参数

```
--dataset           评测数据集路径（默认 evals/dataset.example.jsonl）
--mode              融合策略：weighted / rrf / semantic / bm25
--alpha             weighted 模式语义权重（0.0~1.0）
--rrf-k             RRF 常数（默认 60）
--bm25-top-k        BM25 召回数（默认 20）
--vector-top-k      向量召回数（默认 20）
--top-k             最终保留数（默认 5）
--semantic-threshold 语义阈值（默认 0.0）
--rerank / --no-rerank  是否启用精排（默认启用）
--rerank-top-n      精排保留数（默认等于 --top-k）
--out-dir           结果输出目录（默认 evals/results）
```

### 1.3 参数速查表

| 参数 | 默认值 | 作用 | 调优建议 |
|------|--------|------|---------|
| `mode` | `weighted` | 选择检索融合策略 | 先用 weighted 基线，再试 rrf 对比 |
| `alpha` | `0.5` | 语义权重占比 | 中文短问题偏向 0.3~0.5；长句偏向 0.6~0.8 |
| `rrf_k` | `60` | RRF 常数 | 通常不需要改，保持 60 |
| `bm25_top_k` | `20` | BM25 初召回 | 文档少时可降到 10 |
| `vector_top_k` | `20` | 向量初召回 | 同上 |
| `final_top_k` | `5` | 送 LLM 的片段数 | 3~8 间，太多会稀释上下文 |
| `semantic_threshold` | `0.0` | 语义过滤阈值 | 噪声多时调到 0.3~0.5 |
| `rerank_enabled` | `true` | 是否 LLM 精排 | 精排可明显提升首位命中率，但增加延迟 |
| `rerank_top_n` | `5` | 精排后保留数 | 通常 ≤ final_top_k |

## 二、完整测试流程

从零开始，走完一次完整的系统测试。

### 2.1 环境准备

```powershell
# 1. 安装 Python 依赖（使用 uv）
cd "项目根目录"
uv sync

# 2. 安装前端依赖
cd frontend
npm install
cd ..

# 3. 准备 LLM
#    方式 A：启动 LM Studio，加载对话模型 + 嵌入模型，监听 localhost:1234
#    方式 B：在管理后台配置远程 API（如 DeepSeek / 智谱）
```

### 2.2 启动系统

```powershell
# 一键启动（Windows）
.\scripts\start-dev.ps1

# 或分别启动
# 终端 1：uv run uvicorn api:app --host 0.0.0.0 --port 8000
# 终端 2：cd frontend && npm run dev -- --port 3000
```

启动后：
- 后端 API → http://127.0.0.1:8000
- 前端页面 → http://localhost:3000
- API 文档 → http://localhost:8000/docs

### 2.3 上传文档建立知识库

1. 打开 http://localhost:3000，用 `admin` / `123456` 登录
2. 进入「文档管理」页面
3. 拖拽文件到上传区域（支持 PDF / DOCX / TXT / 图片等）
4. 等待处理完成（系统自动完成：文本提取 → 切片 → 向量嵌入）
5. 可在「向量片段」页面检查切片结果

### 2.4 对话测试

1. 回到首页（对话页）
2. 点击「新建对话」
3. 输入问题，观察：
   - 回答是否基于文档内容（参考来源是否正确）
   - 流式输出是否流畅
   - 多轮对话上下文是否连贯
4. 尝试切换不同模型（顶栏下拉菜单）
5. 尝试关闭「参考文档」开关，对比纯 LLM 回答

### 2.5 检索质量评测

#### 方法 A：UI 调参预览

1. 进入 `/admin` → 「检索调参」标签页
2. 在右栏输入测试问题，点击检索
3. 检查召回片段是否正确
4. 切换策略 / 调整参数 / 再次检索对比
5. 满意后「保存到全局」

#### 方法 B：命令行批量评测

```powershell
# 编写自己的评测集
# evals/my_dataset.jsonl（内容根据你上传的文档编写）

# 对比跑四种策略
uv run python scripts/eval_retrieval.py --dataset evals/my_dataset.jsonl --mode weighted --alpha 0.5
uv run python scripts/eval_retrieval.py --dataset evals/my_dataset.jsonl --mode weighted --alpha 0.7
uv run python scripts/eval_retrieval.py --dataset evals/my_dataset.jsonl --mode rrf
uv run python scripts/eval_retrieval.py --dataset evals/my_dataset.jsonl --mode semantic --no-rerank

# 查看结果
ls evals/results/
```

### 2.6 端到端验证清单

| # | 测试项 | 操作 | 预期结果 |
|---|--------|------|---------|
| 1 | 用户注册/登录 | 注册新用户 → 登录 | 成功跳转到对话页 |

## 三、技能管理

### 3.1 什么是技能

**技能（Skill）** 是一组预设的 **系统提示词 + 自动识别规则**。切换技能相当于让同一个 LLM 扮演不同的专业角色。

```
用户提问 → 检索知识库 → 技能提供的 system_prompt → LLM 生成回答
                                      ↑
                         不同技能 = 不同角色身份 + 不同回答风格
```

### 3.2 系统预置技能

首次启动时自动创建以下技能：

| 技能 | 图标 | 用途 | 自动识别关键词 |
|------|------|------|---------------|
| 财务分析 | 📊 | 财报解读、利润分析、资产负债 | 财务、报表、利润、资产、负债、营收、成本 |
| 会议纪要助手 | 📝 | 会议记录整理、议题提取、待办追踪 | 会议、纪要、议程、决议、讨论、参会、待办 |
| 本地政策文档 | 📋 | 政策法规查询与条款解读 | 政策、规定、条例、办法、通知、制度 |
| 发票助手 | 🧾 | 发票信息提取、金额核验、报销明细 | 发票、invoice、税额、价税、开票、报销 |

### 3.3 创建自定义技能

#### 通过前端 UI 创建

1. 用管理员登录，进入 `/admin` 页面
2. 切换到 **「技能管理」** 标签页
3. 点击右上角 **「+ 新建技能」** 按钮
4. 填写表单：

| 字段 | 是否必填 | 示例 |
|------|---------|------|
| 名称 | 必填 | 法律合同助手 |
| 图标 | 选填 | ⚖️ |
| 描述 | 选填 | 合同条款解读与风险提示 |
| 系统提示词 | 必填 | 你是一个专业的法律顾问。请根据参考资料... |
| 规则 | 选填 | 关键词=回答（每行一条） |
| 自动识别关键词 | 选填 | 合同, 条款, 甲方, 乙方, 违约 |

5. 点击「保存」

#### 系统提示词编写示例

**示例 1：法律合同助手**

```
你是一个专业的法律合同审查助手。请根据参考资料，用严谨的中文回答用户关于合同条款、
权利义务、违约责任等方面的问题。

回答要求：
- 引用具体条款编号和原文
- 对比不同合同中的差异
- 标注潜在风险点
- 必要时给出法律依据
```

**示例 2：技术文档助手**

```
你是一个技术文档阅读助手。请根据参考资料，回答用户关于 API 接口、配置参数、
部署步骤等技术问题。

回答要求：
- 给出准确的配置项名称和取值
- 包含代码示例（如有）
- 注明版本差异
- 对步骤类问题给出编号列表
```

### 3.4 使用技能

技能的使用有两种方式：

#### 自动识别（被动）

上传文件时，系统根据**文件名**匹配技能的自动识别关键词：

```
上传「2024年度审计报告.pdf」
  → 文件名包含「审计」→ 不匹配
  → 文件名包含... 如果你把「审计」加到了「财务分析」的关键词中
  → 页面顶部弹出横幅：「检测到财务相关文档，是否切换到财务分析技能？」
  → 点击确认 → 后续对话使用财务分析的系统提示词
```

#### 手动切换（主动）

1. 在对话页面，查看侧栏底部的 **「技能场景」** 区域
2. 点击想要使用的技能卡片
3. 新建对话后，该对话将使用所选技能的系统提示词

> 技能切换仅影响**新建**的对话。已有对话保持创建时的技能设定。

### 3.5 技能字段说明

| 字段 | 用途 | 详细说明 |
|------|------|---------|
| **名称** | 展示标识 | 显示在侧栏和管理后台 |
| **图标** | 视觉标识 | 支持 emoji，如 📊📝⚖️🔬 |
| **描述** | 简要说明 | 展示在技能卡片上 |
| **系统提示词** | 核心内容 | 作为 LLM 的 system message，定义角色和回答风格 |
| **规则** | 直答规则 | 格式 `关键词=回答`，命中后跳过 LLM 直接返回（零延迟） |
| **自动识别关键词** | 文档匹配 | 多个关键词用逗号或换行分隔，用于上传文档时自动推荐技能 |

### 3.6 技能最佳实践

1. **系统提示词要具体**：不要只写"你是一个助手"，要写清角色身份、回答格式、注意事项
2. **规则用于高频问答**：把确定性很高的 FAQ 写成规则，响应速度从秒级降到毫秒级
3. **自动识别关键词要精准**：太泛（如"文件"）会误触发，太窄（如"2024年度预算执行报告"）会漏匹配
4. **一个技能聚焦一个领域**：不要把财务和法律混在一个技能里，分开更精准
5. **迭代优化提示词**：先用简单提示词上线，根据实际对话效果逐步完善
